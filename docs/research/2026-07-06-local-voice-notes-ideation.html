Grounding Context

Topic Context

Shape. Two core loops: voice capture → local transcription → durable storage, and later recall/reference. Delivery: Python backend + TypeScript web UI. Usage: on-demand — the user activates it to capture and returns to it to recall; nothing needs to run continuously.

Hard constraints. Local models only (no cloud dependency for core function); on-demand operation; Python backend; TypeScript frontend where needed; capture and recall are both first-class; personal single-user tool. Polish ("nice web UI", "nice system") is an explicit requirement, not decoration.

What the brief leaves open (the surface these ideas explore): which local STT model and hardware path; storage format; enrichment beyond the raw transcript; the recall interaction model (search-first vs browse-first vs timeline-first); error correction; durability/backup; single web app vs background service vs hybrid.

External Research (fresh, 2026-07-06 — research value: high)

  • The market splits capture from memory. Excellent local capture tools exist (VoiceInk, superwhisper-class dictation apps) with no recall; the best capture+recall product (Voicenotes) is cloud-only and publicly criticized for weak search (synonym misses, no boolean/tag scoping, related-notes breaking on long notes). Nobody ships local capture plus real recall.
  • Local recall has a convergent pattern. SQLite FTS5 (precision) + sqlite-vec (semantic recall) + Reciprocal Rank Fusion — "no server, single file" — is independently recommended by multiple credible sources.
  • The model landscape churns fast. Parakeet TDT v3 beats Whisper large-v3 on accuracy at half the size; Voxtral (Apache 2.0) hits 2.1% WER; Moonshine targets streaming at ~6× smaller; on Apple Silicon the frontier is shifting to CoreML/ANE (FluidAudio ~110× realtime), which is Swift-first — a real tension with a Python backend.
  • Known reliability gap. Whisper-family models hallucinate on silence/noise (~1% of transcripts, looping phrases); mitigations are VAD segmentation (WhisperX-style) or anti-hallucination-trained models (Parakeet).
  • Structural cautionary tale. Rewind AI pitched "your data never leaves your device," pivoted, was acquired by Meta (Dec 2025), and killed its Mac app with two weeks' export notice. For a personal memory tool, "local" is not enough — the data must outlive the app.

Grounding notes: no institutional learnings corpus exists (greenfield); no user-supplied research artifacts; grounding = the brief + fresh web research below.

1

The archive outlives the app

architecture-form-factor basis: external confidence 90% complexity Medium
Description
Make the data plane app-independent and re-derivable. Ground truth is a plain folder: one directory per note holding the original audio (immutable, content-addressed) and a human-readable Markdown transcript with YAML frontmatter carrying generous capture metadata (timezone-aware timestamp, trigger, duration, links). SQLite — FTS5 + sqlite-vec — is a derived, rebuildable index, never the source of truth; rebuild-index regenerates the whole recall layer from the folder. Every derived artifact (transcript, embedding, tags, summary) is stamped with the model, version, and parameters that produced it, and a first-class re-derive command batch re-transcribes and re-embeds the corpus when a better local model ships — with a diff review before the index rebuild. Park the folder in any synced location (iCloud Drive, Dropbox, git) and backup/export cease to exist as features: the native format is the export. Optionally enforce the local-only promise structurally by denying the runtime outbound network access, so egress fails loudly instead of creeping in through a dependency.
Basis
external: Rewind AI's entire pitch was "your data never leaves your device"; it pivoted, was acquired by Meta (Dec 2025), and killed its Mac app with two weeks' export notice — the research digest calls this structurally decisive for personal local-first tools. FTS5 + sqlite-vec + RRF is the convergent "no server, single file" local pattern (sqlite-vec author, Simon Willison, sqlite.ai independently). Model churn is documented: Voxtral at 2.1% WER, Parakeet TDT v3 beating Whisper large-v3 at half the size, Apple Silicon engine rankings "moving fast."
Rationale
For a personal memory tool, the corpus is the asset and the app is disposable tooling around it. Audio is the only non-depreciating artifact — any transcript derived today is the worst it will ever be — so the archive should appreciate as models improve rather than fossilize at 2026 quality. This single decision de-risks every other idea in this set: features can be rebuilt; the data never has to be.
Downsides
Write-path discipline costs complexity (folder + markdown mirror + index kept consistent). Lossless audio grows storage, though deliberate on-demand notes stay small. The re-derive flow needs diff-review UX to be trustworthy. Egress-deny packaging (sandbox profile) adds friction and can wait for a later phase.
2

Split the lifecycle: an armed capture sliver + an ephemeral reading room

architecture-form-factor basis: external + direct confidence 80% complexity High
Description
One core, two lifecycles, everything behind a contract. A Python core (FastAPI-class) owns capture and recall logic behind a stable local API with an explicit on-demand lifecycle; the TypeScript web UI is just its first client — the reading room that boots on demand and dies on close. Capture gets its own surface: a minimal resident trigger (menu bar / global hotkey) keeping a small STT model warm so keypress→recording feels instant and never requires a browser tab. Below the core, STT engines are swappable sidecar processes behind an OpenAI-compatible transcription contract — whisper.cpp or MLX on day one, a Swift FluidAudio CoreML/ANE sidecar when the speed matters — with a built-in bench command that races installed engines on your own recorded voice to pick defaults.
Basis
external: VoiceInk (3.7k★) proves the system-wide-hotkey ergonomic and is "capture-excellent, recall-absent"; Whishper proves the Python-backend + web-UI shape but stays browser-bound. The digest's Apple Silicon head-to-head — fluidaudio-coreml 0.19s < parakeet-mlx 0.50s < mlx-whisper 1.02s < whisper.cpp 1.23s, with the frontier "shifting toward CoreML/ANE… Swift-first, not Python-first" — is the one piece of evidence that cuts against a pure-Python engine; a sidecar contract dissolves that tension without giving up the Python backend.
direct: the brief names two separate on-demand moments — "used on demand to capture notes through voice" and "recall through a nice system that can be used on demand and when needed" — and hedges the frontend ("potentially where needed… TypeScript"), signalling surfaces will vary.
Rationale
"Zero-friction capture" and "nice web UI" pull in opposite directions if forced into one process — a browser tab you must open is where fleeting thoughts die. Splitting lets each half be excellent at its one job, and the API-first core makes every future surface (CLI, phone-as-mic PWA over the LAN, chat client) an additive thin client. The engine contract turns the fastest-moving layer of the stack into a config change and settles the Python-vs-ANE tension in one explicit decision instead of five implicit ones.
Downsides
The most complex survivor — three deliverables (sliver, core, web UI) even in minimal form. Engine-sidecar indirection is overhead if you only ever run whisper.cpp. An optional RAM-only pre-roll buffer (catching the sentence spoken while reaching for the hotkey) stretches "on demand" further — if built at all, it must be opt-in with a visible never-persisted indicator.
Named fork — decide explicitly
The resident sliver deliberately relaxes strict "nothing resident." The alternative is the pure appliance: one stt command cold-boots core + UI, the model lazy-loads on first record, and nothing survives closing the tab — maximally calm, but first capture is slower. Both are honest readings of "on demand." A staged path works: start pure-appliance, add the sliver only when capture latency actually annoys.
3

The consolidation heartbeat: draft instantly, deepen at the seams

transcription-engine basis: reasoned + external confidence 85% complexity Medium
Description
A two-phase pipeline modeled on memory consolidation. At capture, a small fast model (Moonshine / distil-whisper class) puts a readable draft on screen at speaking speed. Everything heavy runs later as one consolidation pass at the natural seams of on-demand use — session end, next app open, an optional idle window: heavyweight re-transcription (large-v3 / Parakeet / Voxtral class) with VAD segmentation and a hallucination sweep, embeddings, auto-title/tags/summary, related-note linking ("you said something different about this in March"), and personal-lexicon application. Each note visibly shows its state (draft → consolidated), so the system is honest about which quality tier you're reading.
Basis
reasoned: instant capture and expensive integration cannot share the same moment — and an on-demand tool has no daemon schedule to lean on, so heavy compute must attach to moments the user already initiates (capturing, opening). Splitting draft from consolidation is the only architecture that keeps capture instant and archive quality high under those two constraints.
external: the digest supplies both tiers — Moonshine matches large-v3 English at ~6× smaller; distil-whisper gives ~99% of large-v3 quality at ~6× speed; Parakeet's anti-hallucination training and WhisperX-style VAD address the documented ~1% silence-hallucination failure.
Rationale
Dissolves the fast-vs-accurate model debate by refusing the premise (both, staged), turns the on-demand constraint into the design's rhythm rather than its limitation, and gives the other survivors — re-derive epochs, recall-on-write, lexicon application — one natural place in the runtime to live.
Downsides
Two models on disk (a few GB). Notes exist briefly in draft state, so search may run on draft text until consolidation lands. The pass must be background and interruptible — it can never block the user — and draft→final text changes need non-jarring presentation (receipts from idea 5 help).
4

Search you can see: exact, semantic, fused — never a black box

recall-retrieval basis: external confidence 90% complexity Medium
Description
Build recall on SQLite FTS5 (exact, boolean, tag-scoped) plus sqlite-vec (chunk-level semantic similarity with local embeddings — BGE-M3 / EmbeddingGemma class) fused via Reciprocal Rank Fusion — and make the retrieval mode a visible, one-keystroke toggle (exact / semantic / fused) rather than one opaque blended box. Every result explains why it matched (term hit vs semantic neighbor). Embeddings carry model provenance so upgrading the embedder is a background re-embed job, not a redesign.
Basis
external: three independent sources converge on FTS5-for-precision + vector-for-recall + RRF as the standard embeddable local pattern ("no server, single file"). The closest commercial analog (Voicenotes) was publicly criticized for exactly the failures this design dodges: keyword search missing synonyms ("headache" vs "migraine"), no boolean or tag-scoped queries, and cosine-similarity related-notes breaking on long notes — gaps the digest calls consistently requested and consistently missing across incumbents.
Rationale
This is the most concretely documented pain in the researched landscape, and the fix is a well-converged single-file pattern — the cheapest place for this system to be decisively better than every incumbent from day one, and the layer the brief's "that I can refer to" ultimately rests on.
Downsides
Hybrid tuning (RRF weights, chunk size) needs light evaluation discipline. Three visible modes cost UI polish — the "nice UI" bar applies. The embedding pass adds time to consolidation (idea 3 gives it a home).
5

Receipts all the way down: layered transcripts you can always audit

enrichment-structure basis: external confidence 85% complexity Medium
Description
Store every note as attributed layers over immutable ground truth: raw ASR output (with per-span confidence), local-LLM cleanup, and human edits — plus derived artifacts (title, summary, action items, tags) as versioned annotations that each point to the exact transcript span and audio timestamps they came from. The UI consequences: a blame toggle on any span ("who wrote this — the model, the LLM, or me?"); AI-generated text always visually distinct and clickable back to the seconds of speech behind it; low-confidence or hallucination-suspect spans rendered honestly instead of with unearned authority. This layering is also what makes a prose-first reading experience safe: the default view can be a cleaned, journal-like rendering — verbatim and audio sit one click beneath.
Basis
external: Granola's trust pattern — AI-added text visually distinguished and hyperlinked to the transcript timestamp it came from — is named in the digest as the portable pattern for trustworthy recall. Whisper-family hallucination (~1% of transcripts, looping phrases) makes uniform, unattributed text actively dangerous in a memory system. AudioPen's popularity proves people prefer the prose rewrite of their rambling — but nobody pairs that legibility with durable, auditable recall.
Rationale
Recall is only valuable if you believe what you re-read. Months later, the summary may be the only thing you look at — "did I actually say that?" must be answerable in one click, with the audio as the court of appeal. Layered provenance is a schema decision that cannot be retrofitted onto flat text, which is why it ranks above bolder features: it has to be right early.
Downsides
Real schema and write-path complexity (spans, layers, annotation versions). Tight receipts need word-level alignment (WhisperX-class). The UI needs restraint so layers inform rather than clutter — blame is a toggle, not the default view.
6

Threads, not a feed: chunks are the atoms, topics are the documents

recall-retrieval basis: reasoned + external confidence 70% complexity High
Description
Borrow medicine's problem-oriented record (Weed's POMR). The storage atom is the topic-coherent chunk — VAD plus local-LLM topic-shift segmentation of each recording, with per-chunk audio spans, confidence, and embeddings — and recordings are just containers. The home surface is a problem list: a persistent, user-confirmed registry of ongoing threads (projects, people, decisions, open questions). Consolidation indexes each new chunk against the registry; opening a thread assembles everything you've ever said about it, chronologically, each span linked to its source audio. A rambling ten-minute session feeds three threads instead of poisoning one embedding.
Basis
reasoned: clinicians faced structurally identical conditions — episodic voice-dictated captures accumulating for years, queried later by concern, never by date — and the problem-oriented record has been the standard answer for ~50 years. The query shape transfers exactly: "what did I decide about the renovation," never "what did I say on May 3." A small user-confirmed registry is a controlled vocabulary — the thing incumbents keep shipping without.
external: note-level cosine-similarity "breaks on long or bilingual notes" is named in the research as a category-wide unresolved gap, and chunk-level indexing is the documented direction; auto-tagging and tag-scoped queries are "consistently requested and consistently missing."
Rationale
The brief's "that I can refer to" gets answered structurally: browse-first recall that compounds with every capture, instead of a retrieval gamble against a reverse-chronological feed. This is the recall paradigm bet — the survivor that would most change what the product is.
Downsides
The boldest bet in the set: topic-segmentation quality gates everything (bad splits are worse than none); registry upkeep is a small ongoing chore and must stay optional; the thread UI is real product work; and the pattern is unproven at personal (vs clinical) volume — hence the lowest confidence in the set. Ship search (idea 4) first; grow threads on top.
7

Corrections compound: fix it by saying it again — once

capture-flow basis: reasoned confidence 75% complexity Medium
Description
Make error repair voice-native, and make every repair permanent knowledge. Punch-in correction, borrowed from music production: select a botched span and re-speak it; the engine re-transcribes just that span conditioned on the surrounding accepted text and your personal lexicon, and prior takes are kept (comping — a correction never destroys what was actually said). Every accepted fix immediately patches the transcript and its search-index entries, and is mined into a personal lexicon of your proper nouns, projects, and jargon — injected into future cleanup passes, offered retroactively as suggested fixes across the archive, and used by the engine bench to measure accuracy on your voice specifically. Fix "Shuaib" once; it stops being mangled everywhere, forever.
Basis
reasoned: a single-user corpus has a small, stable set of recurring rare tokens — names, projects, domain terms — which is exactly where ASR errors concentrate (even the best local models run ~6% word error rate per the research). An uncorrected error is not cosmetic: it silently removes that note from being findable by its own correct terms, breaking the system's core promise for precisely the note it occurs in. And keyboard-only correction re-introduces, at the exact moment of failure, the typing friction voice was chosen to eliminate.
Rationale
No tool in the researched landscape offers per-user adaptation. This converts the most annoying chore in any STT tool into the mechanism that makes this one the most accurate STT system in existence for a single person: you.
Downsides
Span-targeted re-transcription with clean alignment is fiddly engineering. Lexicon application must suggest, never silently rewrite (the receipts from idea 5 make this safe). Voice-targeting a span ("fix the part about…") is a stretch goal — click-to-select is the honest v1.
